### 2. 我的错误诊断与避坑指南

今天你的代码主体逻辑其实都非常正确，卡住你的主要是**C++ 语法细节**和**编译环境兼容性**。以下是你需要特别标记的“雷区”：

- **致命拼写失误**：
    
    - 入口函数 `mian()` 绝对无法链接，必须是 `main()`。
        
    - 标准库函数少字母，如 `next_permuation`，编译器不会宽容任何拼写错误。
        
- **条件分支语法红线**：
    
    - `else` 后面**严禁直接加括号写条件**。必须写成 `else if (条件)` 或单独的 `else { ... }`。写错会导致连环报错（如 `expected ';' before '{' token`）。
        
- **版本兼容性黑洞（C++11 vs C++17）**：
    
    - 在算法竞赛（如蓝桥杯 C++11 环境）中，`<numeric>` 库里**没有** `std::gcd`。
        
    - **避坑方案**：永远准备好手写极简版的辗转相除法 `gcd`，或者使用 GCC 专有的 `__gcd(a, b)`。
        
- **全排列模版的唯二铁律**：
    
    - **忘写排序**：调用 `next_permutation` 前，必须 `sort` 数组。否则无法从最小字典序开始，会漏掉大量解。
        
    - **误用 while 循环**：必须使用 `do...while`。如果直接用 `while (next_permutation(...))`，会因为函数先执行并修改数组，导致直接漏掉初始的第一个排列。

### 3. 更优解法与复杂度对比

今天我们探讨的题目都存在“暴力解”与“优化解”的对比。

**关于“烤鸡”题（DFS）：**

- **你的原始思路**：纯暴力 $O(3^{10})$ 搜到底，虽然能在 10 的规模下通过，但如果规模扩大就会 TLE（超时）。
    
- **更优解法（可行性剪枝）**：在 DFS 内部加入预判。如果 `当前和 + 剩余配料全放 1 克 > 目标值` 或 `当前和 + 剩余配料全放 3 克 < 目标值`，直接 `return`。
    
- **为什么更好**：这能将搜索树的无效分支提前斩断，时间复杂度成指数级断崖下降。
    

**关于“免费披萨”题（全排列）：**

- **常见错误替代思路**：逆向推导，先求总数 $n$ 的因数，然后去验证倍数是否为合法订单。
    
- **你的全排列思路（更优）**：正向枚举生成。
    
- **为什么更好**：逆向推导的时间复杂度极不稳定（最坏 $O(n)$，验证千万级别数据必超时）。而正向使用 `next_permutation` 的时间复杂度被严格封顶在 $O(K \log V)$（本题最多执行 $\sim 1.4 \times 10^6$ 次循环），且空间复杂度为极优的 $O(1)$。

#### 模版三：通用 DFS 回溯框架

适用于需要在搜索半途中停下、或者状态复杂的组合问题。

C++

```
void dfs(int step, int current_state, int target) {
    // 1. 剪枝优化（预判死胡同，直接回头）
    if (/* 预判当前状态已经不可能达到目标 */) return;

    // 2. 结束条件（到达叶子节点）
    if (step == 目标深度) {
        if (current_state == target) {
            // 记录当前正确路径
        }
        return;
    }

    // 3. 遍历当前步骤的所有可能选择
    for (int i = 1; i <= 3; ++i) { // 假设有 3 种选择
        // 做出选择（压入路径、更新状态）
        path.push_back(i);
        
        // 带着新状态进入下一层递归
        dfs(step + 1, current_state + i, target);
        
        // 撤销选择（回溯核心：恢复现场）
        path.pop_back();
    }
}
```
