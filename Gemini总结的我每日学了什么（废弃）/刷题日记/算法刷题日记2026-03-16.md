### 模块一：今日知识清单与特定触发机制

- **1. 深度优先搜索 (DFS) 与回溯**
    
    - **Trigger (触发条件)**：题目出现“求所有方案数”、“输出所有合法路径/排列组合”、“寻找一条满足所有条件的路径”，且数据范围极小（通常 $N \le 20$）。第一反应必须是：画解答树，用 DFS 暴力搜，走不通就回溯（撤销状态）。
        
- **2. 集合查重 (`std::set` / `std::unordered_set`)**
    
    - **Trigger (触发条件)**：当题目要求“不能经过重复状态”，且你的状态是一个数组 (`vector`) 或字符串时。第一反应：不要用 `vector` 去遍历查找，直接把状态扔进 `set`，用 `vis.count(state)` 做到 $O(\log N)$ 或 $O(1)$ 的查重。
        
- **3. 状态压缩 (用整数代替数组)**
    
    - **Trigger (触发条件)**：题目具有“仅包含两种状态（如选/不选、空/满）”的特征，且总物品/位置数 $N \le 30$。第一反应：不要开 `vector<int>`，直接用一个十进制整数表示状态，利用除法/取模（或位运算）提取位信息，大幅降低空间消耗。
        
- **4. 格雷码 (Gray Code)**
    
    - **Trigger (触发条件)**：看到“每次只能改变一个二进制位/一个物品状态”、“求 $N$ 维超立方体哈密顿回路”、“遍历 $2^n$ 种所有可能组合且相邻状态只差一步”。第一反应：抛弃搜索，直接套用 $i \oplus (i \gg 1)$ 模版。
        
- **5. 记忆化搜索 (DFS 的进阶 DP)**
    
    - **Trigger (触发条件)**：用 DFS 求方案总数时，发现到达了相同的“剩余资源状态”，且数据范围让纯 DFS 必定超时（如 $10^6$ 级别以上）。第一反应：建一个多维数组（小本本），存下每个状态的返回值，搜过一次直接查表。

### 模块二：傻瓜式解题框架（思路提取）

针对今天的**“组合分配/路径搜索类 DFS”**，你的大脑需要建立以下四步思考路径：

1. **第一步：明确状态与起点 (初始化)**
    
    - 问自己：我需要几个参数来描述当前的进度？（例如：当前到了第几个人 `st`？手里还剩多少资源 `a`, `b`？）
        
    - 起点传入什么？（例如：第 0 个人，已发资源 0, 0）。
        
2. **第二步：明确终点 (写在函数最开头)**
    
    - 问自己：怎样才算走到头？（例如：人数 `st == 7` 或者步数 `num == total`）。
        
    - 走到头之后，还需要满足什么额外条件才算“成功”？（例如：资源必须刚好等于目标值 `a == 9 && b == 16`）。满足就记录答案，**必须 return**。
        
3. **第三步：寻找下一步与剪枝 (核心循环)**
    
    - 问自己：当前阶段（针对当前这个人/当前这个状态），我有哪些合法的选择？
        
    - 写出 `for` 循环遍历所有选择（注意起点是 0 还是 1！能否什么都不选？）。
        
    - **剪枝**：如果发现资源已经超发（`a > 9`），立刻 `return` 砍掉这条分支。
        
4. **第四步：深入与回溯 (做选择 -> 往下走 -> 撤销)**
    
    - 把做出的选择累加到参数中，递归调用下一层（注意是加具体的值 `a + i`，绝不是仅仅 `a + 1`）。
        
    - 如果需要防走回头路，进去前加 `vis.insert`，出来后必须 `vis.erase` 恢复现场。

### 模块三：我的源码级纠错与降维打击（重点！）

#### 1. The Rock Game - DFS 尝试 (原味纠错)

C++

```
void dfs (int st,int num)
{
    if(arr.size()>n)return; 
    
    // 错误点1：逻辑不严密。num==0 并不能证明你收集齐了 2^n 个互不相同的状态。
    // 你需要一个目标总数 tal = 1 << n，判断 num == tal。
    if(num==0&&st!=0)
    {
        test.push_back(arr);
        return;
    }

    arr.push_back(0);
    // 错误点2：致命陷阱！st++ 和 num++ 是后缀操作。
    // 这意味着传入函数的依然是原来的 st 和 num，函数内部会无限递归导致爆栈。
    // 必须写成：dfs(st + 1, num);
    dfs(st++,num); 
    arr.pop_back();
    // ...
}
```

#### 2. 分糖果 - 按照人分配 (原味纠错)

C++

```
void dfs(int st,int a,int b)
{
    // ... 终点判断省略 ...

    // 错误点1：隐藏条件预设错误。
    // 你把 i 和 j 的起点设为了 1，强迫每个小朋友至少拿 1 颗 a 和 1 颗 b。
    // 但题目只规定“总数”在 2~5，并没有规定单种糖果不能为 0。
    // 纠正：必须改成 int i = 0; 和 int j = 0;
    for(int i = 1;i<=9-a;++i) 
    {
        for (int j = 1;j<=16-b;++j)
        {
            if(i+j>=2&&i+j<=5)
            {
                // 错误点2：逻辑断层。
                // 循环里你决定给这个小朋友发 i 颗 a 糖果，
                // 结果传递给下一层时，你却硬编码写成了 a+1。
                // 这导致无论你发几颗，系统都以为只发了 1 颗，数据永远对不上。
                // 纠正：dfs(st+1, a+i, b+j);
                dfs(st+1,a+1,b+j); 
            }
        }
    }
}
```

#### 🚀 降维打击专区

**针对题目：The Rock Game (P6183)**

- **你的原思路**：用 DFS 穷举所有可能性，用 `set` 查重寻找哈密顿路径。时间复杂度最坏 $O((2^n)!)$，空间复杂度 $O(N \cdot 2^n)$。
    
- **极其明显的替代算法：格雷码 (Gray Code)**
    
    - **好在哪里**：这道题本质就是求一个二进制序列，要求相邻两个数只有一位不同。格雷码完美契合这个数学规律。
        
    - **为什么更优**：完全抛弃了“搜索”和“试错”的过程。只需要一个 $O(2^n)$ 的 `for` 循环，利用公式 $G(i) = i \oplus (i \gg 1)$（或对应的加减法逻辑）直接**按顺序打印答案**。空间复杂度降为绝对的 $O(1)$，时间复杂度压缩到了物理极限。

### 模块四：核心代码模版与保姆级拆解

今天最有价值的通用模版是**“分配组合类 DFS”**（以修复后的分糖果代码为例）：

#### 1. 完整代码

C++

```
#include <bits/stdc++.h>
using namespace std;

int res = 0;

void dfs(int st, int a, int b) {
    if (st == 7) {
        if (a == 9 && b == 16) {
            res++;
        }
        return;
    }

    if (a > 9 || b > 16) return;

    for (int i = 0; i <= 9 - a; ++i) {
        for (int j = 0; j <= 16 - b; ++j) {
            if (i + j >= 2 && i + j <= 5) {
                dfs(st + 1, a + i, b + j);
            }
        }
    }
}

int main() {
    dfs(0, 0, 0);
    cout << res << "\n";
    return 0;
}
```

#### 2. 逐行剖析

- `if (st == 7)`：拦截器。当递归层数达到目标人数时，强制停下，不让程序继续往下越界。
    
- `if (a == 9 && b == 16)`：验尸官。在拦截下来的终点状态中，检查资源是否刚好完美耗尽。是的话才给答案 `res++`。
    
- `if (a > 9 || b > 16) return;`：剪枝利器。一旦发现当前挥霍无度，资源超标了，后面的小朋友就算分出花来也是错的，立刻掐断这条世界线，省下海量运算时间。
    
- `for (int i = 0...` 和 `for (int j = 0...`：枚举机器。把当前小朋友所有合法的拿糖姿势全部列举一遍。
    
- `if (i + j >= 2 && i + j <= 5)`：题意滤网。严格按照题目的特定要求过滤非法姿势。
    
- `dfs(st + 1, a + i, b + j);`：时空跃迁。带着当前更新后的资源总量，把接力棒交给下一个小朋友。
    

#### 3. 参数透视

- `int st`: **阶段指示器**。代表目前正在处理第几个“分配对象”（如第几个小朋友/第几个洞）。**边界注意**：从 0 开始的话，终点判断就是 `st == 目标总数`。
    
- `int a, int b`: **资源蓄水池**。代表从起点到现在，总共消耗了多少资源。**边界注意**：向下传递时，一定要加本层消耗的**变量值**（`i`, `j`），而不是写死常数（`+1`）。
    
- `返回值 void`: 本模版只是为了“遍历并全局计数”，所以不需要返回值。如果是记忆化搜索，就要改成 `int`，返回以当前节点为根的子树里有多少个合法方案。

### 模块五：今日高频盲区与灵魂自测

**高频盲区总结**：

1. **“动作”与“阶段”的混淆**：在用 DFS 解决分配问题时，不能把“发一颗糖”当成进入下一层的标志。进入下一层（`st+1`）必须意味着**当前对象的所有动作已经彻底清算完毕**。
    
2. **函数传参的后缀递增陷阱**：在 C++ 中，`dfs(st++)` 会先提取 `st` 的当前值传入函数，**然后再**把 `st` 加 1。这等同于把一模一样的参数传给了下一层，造成死循环。
    

**灵魂自测（请先遮挡答案，在脑海中作答）：**

**Q1：在 `The Rock Game` 这类求路径的题中，如果我忘了写 `vis.erase(nex)`（撤销访问标记），程序会发生什么？**

  
  
  
  

> **答案**：程序不会死循环，但是会漏掉大量正确解，甚至找不到答案。因为一旦某个状态在某条错误的路径中被访问过，它就被永久封杀了。当你想从另一条正确的路径经过这个状态时，会被 `vis` 墙挡住，导致 DFS 无法完全探索状态空间。

**Q2：在分糖果的循环中 `for(int i = 0; i <= 9 - a; ++i)`，如果我把条件错写成 `i <= 9`，代码会报错吗？会带来什么逻辑后果？**

  
  
  
  

> **答案**：代码本身不会引发语法报错或数组越界，但逻辑会全错。这会导致你在分配时，无视了“剩余糖果量”的限制，试图发出去比仓库里还要多的糖果。虽然外层的剪枝 `if (a > 9) return;` 可能会在下一层把它拦截下来，但这会产生极其庞大的无效搜索树，严重拖慢运行速度。

**Q3：如果分糖果的题目变成“有 100 颗糖，50 个小朋友”，原封不动的 DFS 必定超时。如果我想加入“记忆化小本本 (Memoization)”，需要加哪三行最关键的代码逻辑？**

  
  
  
  

> **答案**：
> 
> 1. 初始化一个小本本：`int memo[55][105][...];` 并且全部填满 -1。
>     
> 2. 函数开头查本子：`if (memo[st][a][b] != -1) return memo[st][a][b];`
>     
> 3. 退出函数前记本子：`memo[st][a][b] = 刚刚算出的合法方案总数;` 并且 `return` 这个值。
