### 一、 Dev-C++ 有 Debug 工具吗？

**有，而且在蓝桥杯比赛中是完全可以、也应该使用的。**

Dev-C++ 内置了基于 GDB 的图形化调试器，它能让你做到“不用写一句 `cout`，就能实时看到所有变量的变化”。

**在蓝桥杯考场上快速开启 Debug 的步骤：**

1. **确认编译选项（关键）**：默认情况下，Dev-C++ 可能没开调试信息。你需要在顶部的下拉菜单中，将编译器配置从 `Release` 改为 `Debug`（例如 `TDM-GCC 4.9.2 64-bit Debug`）。
    
2. **打断点（Breakpoint）**：在你觉得可能出错的代码那一行，点击行号左边的空白灰色区域。会出现一个红色的勾或红点。程序运行到这里就会自动暂停。
    
3. **启动调试**：不要按 F11（编译运行），而是按 **F5（调试）**，或者点击下方控制台的“调试”按钮。
    
4. **添加查看（Watch）**：点击左侧栏的“添加查看”，输入你想追踪的变量名，比如 `r`, `c`, 或者数组项 `mtr[0][0]`。
    
5. **单步执行**：按 **F7（下一步 / Next line）**。代码会一行一行往下走，你可以死死盯着左侧的变量栏，看着它们的值是怎么变化的，瞬间就能抓住是哪一步出了偏差。
    

**⚠️ 考场残酷真相警告：** Dev-C++ 的调试器比较老旧，在处理递归（DFS）层数极深，或者开超大数组时，**非常容易卡死或直接崩溃**。所以在蓝桥杯考场上，**一定要随时按 Ctrl+S 保存代码**再点调试。调试工具是好帮手，但绝不能是你唯一的救命稻草。

### 二、 只有“跟着程序走一遍”这一种笨办法吗？

本质上，找 Bug 确实是寻找“你的预期”和“程序实际走向”的偏差，但你绝对不需要“从头到尾”走一遍。以下是算法竞赛选手常用的高效进阶 Debug 技巧：

#### 1. 二分输出法 (Binary Search Debugging)

既然写 `cout` 慢，那就减少写的次数。把算法里的“二分查找”思维用在查错上。

- **做法**：如果你的程序有 100 行，先在**第 50 行**的中间位置 `cout` 一下关键变量。
    
- **效果**：如果第 50 行输出的数据是对的，说明 Bug 在下半部分；如果错了，说明 Bug 在上半部分。你只需要输出几次，就能把 Bug 锁定在极小的几行代码里，直接将排错时间缩短一半以上。
    

#### 2. 考场极速排错神器：`assert` 断言

与其写 `if-cout`，不如让程序在出错的瞬间自己“自爆”，并告诉你它死在哪了。 在头文件加入 `#include <cassert>`。在你怀疑可能越界的地方，写下你认为**绝对应该成立的条件**：

C++

```
void dfs(int r, int c) {
    assert(r >= 0 && r <= 4); // 我断言 r 绝对不可能超过 4！
    assert(c >= 0 && c <= 4); // 我断言 c 绝对不可能超过 4！
    // ... 你的代码
}
```

**效果**：运行程序，如果 `r` 变成了 `5`，程序会立刻崩溃，弹出一个黑框，上面清晰地写着：`Assertion 'r <= 4' failed.`，并且会**直接标出是在哪一行崩溃的**。这比你用 `cout` 盯着屏幕找快一万倍。提交代码前把 `assert` 删掉或注释掉即可。

#### 3. 建立静态查错清单（瞪眼法）

与其一出错就马上上数据测，不如先停下来，花一分钟用眼睛扫描你的代码。蓝桥杯 80% 的 Bug 都集中在这五个“死亡区域”：

- **多组数据清空**：题目是不是有多组测试用例？你的全局数组和变量有没有用 `memset` 彻底清空？（这一条坑死了无数选手）。
    
- **数组越界**：像咱们刚才那道题，坐标有没有可能等于 `N`？负数坐标有没有处理？
    
- **变量初始化**：局部变量（在函数里声明的变量）有没有赋初值 `0`？
    
- **循环条件**：到底是 `< n` 还是 `<= n`？
    
- **数据溢出**：题目数据范围很大时，累加的过程有没有爆 `int`？是不是该用 `long long`？
    

**总结：** 在接下来的训练中，你可以刻意练习使用 Dev-C++ 的 F5 调试和添加 Watch 的功能，熟悉它的脾气。如果在考场上 Dev-C++ 罢工了，立刻切换到 **二分输出法** 配合 **`assert` 断言**。掌握了这套组合拳，你的 Debug 速度会产生质的飞跃。
