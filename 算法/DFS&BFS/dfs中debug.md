### 1. 考场最强防线：极限状态的“脑内单步执行”

不要去模拟程序一开始是怎么跑的，直接把场景拉到**最后一步**。这是查出越界 Bug 最有效的方法。

- **如何思考**：假设你现在已经顺利填完了整个网格的最后一个格子，也就是第 3 行、第 3 列 `(r=3, c=3)`。根据你的代码逻辑，填完后它会调用 `dfs(3, 4)`。
    
- **代入你的错误代码**：
    
    1. 带着 `r=3, c=4` 进入函数。
        
    2. 第一关 `if(r==4)`：不满足，跳过。
        
    3. 第二关 `if(c==4)`：满足！于是执行 `c=0; r++;`。此时 `r` 变成了 `4`，`c` 变成了 `0`。
        
    4. 继续往下走：代码遇到了 `int v = mtr[r][c];`。
        
    5. **警铃大作**：你立刻意识到它在尝试访问 `mtr[4][0]`，而数组最大只有 `mtr[3][3]`。
        

**总结**：每次写完 DFS 的开头，强迫自己在脑子里只跑一遍 `dfs(最后一行, 最后一列 + 1)` 这个极限状态，90% 的边界越界问题都会在这一秒钟原形毕露。

### 2. 考场暴力 Debug 技巧：入口日志法（Print Debugging）

如果脑内模拟没看出来，或者提交上去是 Runtime Error (RE) / Wrong Answer (WA)，一定要学会让程序自己“说话”。

- **具体操作**：在 `dfs` 函数的**第一行**（任何逻辑之前），无脑加上一句打印：
    
    C++
    
    ```
    void dfs(int r, int c) {
        cout << "Entering dfs: r=" << r << ", c=" << c << endl; // 考场临时加的日志
        // ... 下面是你的原有逻辑
    }
    ```
    
- **如何分析**：运行一下程序。如果是越界导致的错误，在控制台输出的最后几行，你一定会看到类似这样的画面：
    
    Plaintext
    
    ```
    Entering dfs: r=3, c=3
    Entering dfs: r=3, c=4
    Entering dfs: r=4, c=0
    ```
    
    当你看到程序打出了 `r=4, c=0`，并且**还在继续往下执行没有停下来**（或者紧接着就闪退报错了），你就能瞬间定位死穴：结束条件没能拦住 `r=4` 的情况。
    

### 3. 终极解法：固化你的 DFS 状态转移模板

为了避免每次考试都在纠结“谁先谁后”或者“变量是在这层改还是下层改”，最好的方式是**把二维网格的遍历逻辑彻底公式化**。

你现在的做法是：在当前函数里修改传入的参数（`c = 0; r++;`）。这种做法虽然能跑通，但容易产生副作用。

**推荐的通用思考模板（下放责任制）：** 当前层只管当前层的事。如果遇到越界，直接把正确的新坐标打包，踢给“下一层”去处理，同时本层立刻 `return` 结束。

C++

```
void dfs(int r, int c)
{
    // 1. 拦截列越界（最先判断）：如果列到头了，直接把锅甩给下一行的开头
    if(c == 4) 
    {
        dfs(r+1, 0); 
        return; // 甩锅后立刻下班，绝不执行后面的代码
    }
    
    // 2. 拦截行越界（终点）：此时列一定不会越界，安心判断行
    if(r == 4) 
    {
        // 打印答案并退出
        exit(0);
    }
    
    // 3. 处理当前格子的核心逻辑
    if(mtr[r][c] != 0) { ... }
    // ...
}
```

这种写法的优势在于思路极其清晰：第一步纠正坐标，第二步判断终点，第三步干活。各司其职，你以后在考场上哪怕闭着眼睛默写，也不会出现变量被意外修改导致的越界。
