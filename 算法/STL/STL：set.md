### 1. 核心认知 (The Core Logic)

在竞赛中，把 `set` 当作一个**自带魔法的黑盒子**，只要数据扔进去，它自动完成两件事：

1. **天生去重**：同样的数字绝对不存第二份。
    
2. **天生有序**：内部永远保持从小到大的顺序。
    

- **底层机制**：红黑树（一种平衡二叉搜索树）。
    
- **时间复杂度**：增、删、查的速度全都是 $O(\log N)$，极其稳定！
    

---

### 2. 声明与初始化 (Initialization)

- **默认声明 (升序)**：`set<int> s;` (从小到大排)
    
- **降序声明**：`set<int, greater<int>> s;` (从大到小排，经常用到！)
    
- **从 `vector` 一键导入**：
    
    C++
    
    ```
    vector<int> v = {3, 1, 4, 1, 5};
    set<int> s(v.begin(), v.end()); 
    // 瞬间完成去重和排序，s 变成 {1, 3, 4, 5}
    ```
    

---

### 3. 核心操作：增、删、改、查 (CRUD)

#### 3.1 增 (Add)

- `s.insert(x);` (将 `x` 插入集合，如果 `x` 已经存在，则什么都不做，时间复杂度 $O(\log N)$)。
    

#### 3.2 删 (Delete)

- **按值删除 (最常用)**：`s.erase(x);` (把集合里的 `x` 删掉，如果本来就没有 `x`，也不会报错)。
    
- **按迭代器删除**：`s.erase(it);` (删掉迭代器指向的那个位置)。
    
- **清空集合**：`s.clear();`
    

#### 3.3 改 (Modify)

- 🚨 **绝对禁忌：`set` 不能直接修改！**
    
    因为 `set` 的底层是一棵排好序的树，如果你强行把里面的 `3` 改成 `9`，树的结构就崩塌了。
    
    - **正确的修改方法**：先删掉旧的，再插入新的。
        
        `s.erase(3); s.insert(9);`
        

#### 3.4 查 (Query)

抛弃 `<algorithm>`！千万别用 `std::find`！

- ⭐ **找存在感 (count)**：**竞赛中最常用！**
    
    `int cnt = s.count(x);` (因为 `set` 会去重，所以返回值只有两种：`1` 表示存在，`0` 表示不存在)。
```
    if (s.count(5)) { cout << "5 在集合里！"; }
```
- **拿迭代器 (find)**：
    
    `auto it = s.find(x);` (找到了返回指向 `x` 的迭代器，找不到返回 `s.end()`)。
    

---

### 4. 遍历与访问 (Access)

- 🚨 **避坑指南**：`set` **绝对不能**用下标访问（不能写 `s[0]`），只能用迭代器遍历！
    
- **现代 C++ 遍历法 (最推荐)**：
    
    C++
    
    ```
    for (int x : s) {
        cout << x << " "; // 拿出来的顺序天然就是从小到大！
    }
    ```
    
- **获取最大值/最小值**：
    
    - 最小值（首元素）：`*s.begin()`
        
    - 最大值（尾元素）：`*s.rbegin()` (反向迭代器的开头，极其好用的骚操作)
        

---

### 5. 竞赛进阶神技：自带二分查找 (Binary Search)

`set` 内部自带了二分查找函数，极其强大。

- `s.lower_bound(x)`：返回集合中第一个 **大于等于 ($\ge$)** `x` 的元素的迭代器。
    
- `s.upper_bound(x)`：返回集合中第一个 **严格大于 ($>$)** `x` 的元素的迭代器。
    

**🚨 史诗级防坑警告：**

千万不要用 `<algorithm>` 里的 `std::lower_bound(s.begin(), s.end(), x)`！

通用算法面对 `set` 时会退化成 $O(N)$（从头遍历），会导致超时！

**必须写成亲儿子版本：`s.lower_bound(x);`**，这才是 $O(\log N)$。

---

### 6. `set` 的两个亲兄弟 (Variants)

当普通的 `set` 满足不了你时，看看这两个兄弟：

#### 兄弟 1：`multiset` (可重集)

- **特点**：保持从小到大排序，但**允许数字重复**。
    
- **赛场绝技**：常用于动态维护滑动窗口的中位数或最大/最小值。
    
- **危险陷阱**：
    
    `ms.erase(x);` 会把集合里**所有**的 `x` 全部删光！
    
    如果只想删掉一个 `x`，必须写成：`ms.erase(ms.find(x));` (先找到其中一个的迭代器，再删迭代器)。
    

#### 兄弟 2：`unordered_set` (无序集)

- **特点**：天生去重，但**没有任何顺序**。
    
- **底层机制**：哈希表 (Hash Table)。
    
- **速度**：理论上增删改查都是 $O(1)$，极其狂暴。
    
- **赛场评价 (慎用)**：虽然它平时极快，但在某些级别的比赛（如 Codeforces 或天梯高难度题）中，出题人会专门构造“哈希冲突”的数据，让你的 $O(1)$ 退化成极其丑陋的 $O(N)$，导致直接 TLE。所以除非普通 `set` 真的卡时间了，一般优先用安全的 `set`。
---

### 7. `set` 虽好，但不能滥用

既然 `set` 查找这么快，为什么我们平时还是更爱用 `vector` 呢？因为魔法是有代价的：

1. **极度消耗内存**：`set` 里的每个节点除了存数据，还要存左指针、右指针、颜色标识等。存同样多的数字，`set` 占用的内存通常是 `vector` 的 3 倍以上。
    
2. **插入很慢**：`vector` 在尾部加东西 (`push_back`) 是瞬间完成的 ($O(1)$)；而 `set` 每次插入 (`insert`) 都要在树里找位置、调整平衡，速度慢得多 ($O(\log N)$)。
    
3. **不能用下标访问**：你**绝对不能**写 `s[0]` 或者 `s[i]`。`set` 没有下标的概念，你想拿里面的数据，只能用迭代器（`for (int x : s)`）。
    

---

### 💡 判断是否用set

以后在赛场上，该选 `vector` 还是 `set`，你只需要问自己一个问题：**我把这堆数据存下来，主要是为了干嘛？**

- **无脑选 `vector` 的情况：**
    
    - 我只需要把数据存下来，后面按顺序拿出来用。
        
    - 我需要用下标 `v[i]` 来随机访问。
        
    - 我会在最后用一次 `sort` 把它排好序。
        
- **坚决选 `set` 的情况：**
    
    - 题目明确要求**去重**。
        
    - 你需要**非常频繁地查询**“某个数字存不存在”（就像刚刚那道题）。
